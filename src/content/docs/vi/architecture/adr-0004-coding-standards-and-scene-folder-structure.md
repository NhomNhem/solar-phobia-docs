## Bối cảnh

### Vấn đề cần giải quyết
The project needs consistent coding standards enforced via a dedicated namespace, and the scene folder structure has become inconsistent with multiple "Scenes" folders across the project.

### Current Scene Folder Issues
- `Assets/_Project/_Scenes/` — Project scenes (Dev, Gameplay subfolders)
- `Assets/_Project/Settings/Scenes/` — Scene templates for URP settings
- `Assets/Scenes/` — Prototype test scenes (created during development)
- `Assets/Cutscene Engine/Track Examples/Scenes/` — Plugin examples (external)

This duplication creates confusion about where new scenes nên be placed.

### Ràng buộc
- **Engine**: Unity 6000.3.11f1 (Unity 6)
- **Language**: C# 9.0 targeting .NET 4.7.1
- **Architecture**: Clean layering (Domain → Application → Infrastructure/Presentation → Composition)

### Yêu cầu
- Coding standards phải be enforceable in code (not just documentation)
- Scene folder phải be consistent and intuitive for all contributors
- Phải not break existing build configuration

## Quyết định

### 1. Quy chuẩn code Namespace

Create a dedicated `SolarPhobia.Rules` namespace for codifiable coding standards. This namespace will contain:
- Validation rules that can be checked at compile-time or runtime
- Code style enforcers (optional, via analyzers)
- Architectural constraint validators

```csharp
namespace SolarPhobia.Rules {
    // Naming conventions
    public static class NamingConventions {
        public const string DomainNamespace = "SolarPhobia.Domain";
        public const string ApplicationNamespace = "SolarPhobia.Application";
        public const string InfrastructureNamespace = "SolarPhobia.Infrastructure";
        public const string PresentationNamespace = "SolarPhobia.Presentation";
    }
    
    // Validation rules
    public class NoAwakeStartUpdateRuleAttribute : Attribute { }
    public class ConstructorInjectionRequiredAttribute : Attribute { }
}
```

### 2. Cấu trúc thư mục scene

**Primary scene folder**: `Assets/_Project/_Scenes/`

| Subfolder | Purpose |
|-----------|---------|
| `_Scenes/Dev/` | Development and testing scenes |
| `_Scenes/Gameplay/` | Gameplay scenes (main gameplay, levels) |
| `_Scenes/Scenes/` (nested) | Auto-generated by build hệ thống |

**Scene placement rules**:
- All gameplay scenes → `Assets/_Project/_Scenes/Gameplay/`
- All dev/test scenes → `Assets/_Project/_Scenes/Dev/`
- Scene templates (URP) → `Assets/_Project/Settings/Scenes/` (exception for templates)
- Plugin examples → Leave in original locations
- Prototype scenes → Move to `Assets/_Project/_Scenes/Dev/Prototype/` or delete after kiểm tra hợp lệ

**Migration of existing scenes**:
- `Assets/Scenes/Prototype Test Scene.unity` → `Assets/_Project/_Scenes/Dev/Prototype/`

## Các phương án đã cân nhắc

### Alternative 1: Keep multiple scene folders
- **Description**: Allow scenes in multiple locations based on purpose
- **Pros**: Flexibility, matches some existing patterns
- **Cons**: Confusion about where to put new scenes, inconsistent project structure
- **Rejection Reason**: Creates ambiguity for contributors, especially new team members

### Alternative 2: Use Editor-folders for coding standards
- **Description**: Use Unity's Editor-folders (name ending with ~) for rules
- **Pros**: Automatically excluded from build
- **Cons**: Unity-specific, not applicable to C# project standards
- **Rejection Reason**: Need cross-platform, language-level standards, not Unity-specific

### Alternative 3: Only documentation-based standards
- **Description**: Keep coding standards in AGENTS.md only
- **Pros**: Simple, no code required
- **Cons**: Not enforceable, easy to forget, no tooling hỗ trợ
- **Rejection Reason**: Project needs codifiable rules for VContainer/R3 patterns

## Hệ quả

### Positive
- **Clear scene location**: Everyone knows where to put scenes
- **Enforceable standards**: Can build analyzers or validators
- **Cleaner project**: Removes redundant folder structure
- **Better onboarding**: New contributors can quickly understand structure

### Negative
- **Migration effort**: Need to move existing prototype scene
- **Updated documentation**: AGENTS.md phải reference new structure

### Risks
- **Rủi ro**: Build hệ thống scenes (EditorBuildSettings) still reference old paths
  - **Mitigation**: Update Build Settings after migration
- **Rủi ro**: Subscenes (Addressables) reference old folder
  - **Mitigation**: Update Addressable groups after migration

## File Organization Standards

### Services Folder Organization

**Rule**: The `Application/Services/` folder is split into two subfolders to keep interfaces and implementations separate at a glance.

| Subfolder | Contents |
|-----------|----------|
| `Services/Interfaces/` | All `I*.cs` interface files |
| `Services/` | All implementation files (services, hazards, event, data classes) |

**Example structure**:
```
Application/
├── Services/
│   ├── Interfaces/
│   │   ├── IAnimationService.cs
│   │   ├── IAudioService.cs
│   │   ├── IBossSearchlightService.cs
│   │   ├── ICoverDetectionService.cs
│   │   ├── IDayPhaseMechanicsService.cs
│   │   ├── IDayPhaseTimelineService.cs
│   │   ├── IKarmaHazardService.cs
│   │   ├── INightPhaseMovementService.cs
│   │   ├── IPhaseStateMachine.cs
│   │   ├── IWardTimerService.cs
│   │   └── ICharacterController.cs
│   ├── DayPhaseMechanicsService.cs
│   ├── DayPhaseTimelineService.cs
│   ├── KarmaHazardService.cs
│   ├── LuoiMauHazard.cs
│   ├── VungNuocHazard.cs
│   ├── BeDaDaoAnhHazard.cs
│   └── ...
```

**Yêu cầu**:
- All `I*Service.cs` files → `Services/Interfaces/`
- All concrete implementations, MonoBehaviour hazard components, and data classes → `Services/`
- Events (`*Event.cs`) belong in `Application/Messages/`
- Value objects belong in `Domain/ValueObjects/`

### 1 File - 1 Loại

**Rule**: Each type definition (Class, Interface, Enum, Struct, Delegate) phải live in its own dedicated `.cs` file named exactly after the type.

**Examples**:
```
// Assets/_Project/Application/Services/Interfaces/
├── INgocCotService.cs      // Interface for Ngọc Cốt tracking
// Assets/_Project/Application/Services/
├── NgocCotService.cs       // Concrete implementation
// Assets/_Project/Domain/Services/
├── IWardTimerService.cs    // Interface for ward timer (Domain layer)
// Assets/_Project/Application/Services/
├── WardTimerService.cs     // Concrete implementation
```

**Yêu cầu**:
- **Interface files (`I*Service.cs`) → `Application/Services/Interfaces/`**
- **Triển khai files → `Application/Services/`** (grouped with their interface)
- **No multi-type files** — Never combine `interface IBar {}` and `class Bar {}` in one file. Never combine an `enum` with a class.
- **Naming** — File name phải match the type name exactly (case-sensitive).
- **Enforced by**: Code review, future Roslyn analyzer (`SolarPhobia.OneTypePerFileAnalyzer`).

**Rationale**:
- **Layer discipline** — Domain is pure C# (no Unity, no FishNet). Interface in Domain = guaranteed no external dependencies in contract.
- **Faster file lookup** (name = file path)
- **Easier refactoring** (rename type = rename file)
- **Cleaner diffs** (one logical change per file)
- **Prevents "god files"** accumulating multiple types over time
- **Dependency enforcement** — Domain has no upstream dependencies, so keeping interfaces there forces clean architecture. Implementations live in layers that are allowed to depend on Unity/FishNet.

**Anti-patterns**:
```csharp
// ✅ CORRECT — interface in Domain, implementation in Application/Infrastructure
// Domain/Services/INgocCotService.cs     ← Domain layer: pure contract, no Unity/FishNet
// Application/Services/NgocCotService.cs ← Application layer: behaviour
// Infrastructure/Services/WardTimerService.cs ← Infrastructure layer: Unity-dependent

// ❌ WRONG — multiple types in one file
namespace Foo {
    public interface IService { void M(); }
    public class Service : IService { public void M() { } }
    public enum ServiceState { Active, Inactive }
}

// ❌ WRONG — interface and implementation in same layer
// Application/Services/INgocCotService.cs  ← mixes layer concerns
// Application/Services/NgocCotService.cs
```

**Exceptions**:
- Test files may contain multiple fake/test helper classes in one file when they are tightly coupled to a single test class.
- Nested classes within a test file are exempt from this rule (e.g., `TestWardDrainRate` inside `NgocCotRelicPickupsTests.cs`).
